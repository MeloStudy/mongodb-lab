# Laboratory 002: BSON & Data Types Concept

Before attempting the scenario, we need to understand the structural flaw in native primitive data types and how BSON fixes it.

## The IEEE 754 Floating-Point Flaw
In standard JavaScript (and JSON), all numerical values are implicitly treated as double-precision floating-point numbers (IEEE 754). This essentially means the integer `10` and the decimal `10.99` use the same underlying type format: `number`.

This causes the infamous rounding flaw globally known to JavaScript and Python developers:
```javascript
console.log(0.1 + 0.2); // Outputs: 0.30000000000000004
```
If you store financial records with standard JSON numbers, you **will** lose money during aggregations. 

## MongoDB's Solution: BSON (Binary JSON)
MongoDB saves data as BSON, not JSON. While the API *looks* like JSON, BSON includes additional, mathematically-secure data types.

When interacting with MongoDB, we must **explicitly** tell it to map numbers into secure BSON types using specific Wrapper Constructors. If we don't, it defaults everything to a `Double`.

### Core BSON Numerical Wrappers
- **`Int32`** (`NumberInt` in mongosh): A 32-bit integer. Perfect for countable things (user IDs, counters, quantities) to save disk space.
- **`Long`** (`NumberLong` in mongosh): A 64-bit integer, used for massive numbers (like timestamps or huge telemetry IDs).
- **`Decimal128`** (`NumberDecimal` in mongosh): A high-precision 128-bit decimal value. **CRITICAL for financial data, prices, and precise science.**

### The Identity: `ObjectId`
Every document in MongoDB requires a unique `_id` field. By default, MongoDB assigns an `ObjectId`. It is a **12-byte** unique identifier composed of:
1. **Timestamp (4 bytes)**: Represents the seconds since the Unix epoch. This means ObjectIds are *roughly* sorted by time.
2. **Random Value (5 bytes)**: Generated once per process to ensure uniqueness across different machines.
3. **Counter (3 bytes)**: An incrementing value starting with a random value.

**Why use ObjectIds?** 
- They are smaller than UUIDs (12 bytes vs 16 bytes).
- You can extract the creation time directly from them without storing a `created_at` field.
- They are generated client-side by drivers, avoiding server-side bottlenecks.

### The Reality of BinData (Binary Data)
MongoDB can also store raw bytes natively using the `BinData` type. 
This allows you to store:
- Encrypted strings generated by a cipher wrapper.
- Small image thumbnails (e.g., `< 1MB` avatars).
- Hashes or structural telemetry payloads natively.

*Note: The absolute maximum size of a single MongoDB document is **16 Megabytes**. Do not attempt to store MP4 videos or heavy resources using `BinData`, use filesystem architectures like AWS S3 or MongoDB GridFS instead.*
